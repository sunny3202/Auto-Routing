## 2. 클래스 명세

### VoxelGrid

```python
class VoxelGrid:
    def __init__(self, area: BoundingBox, voxel_size: float)
    def mark_obstacle(self, box: BoundingBox) -> None
    def is_blocked(self, voxel_coord: tuple) -> bool
    def world_to_voxel(self, pos: tuple) -> tuple[int, int, int]
    def voxel_to_world(self, coord: tuple) -> tuple[float, float, float]
```

- 공간 범위: `voxel_area_fn()` 반환값
- 복셀 크기: `voxel_size_fn()` = 300mm 고정
- 장애물: `restricted_area_setting_fn()` 필터 후 BIM 장애물 마킹
- 내부 구현: `numpy` 3D boolean array
- 순차 라우팅: 완료된 경로 복셀 → 즉시 `mark_obstacle()` 추가

### PathFinder (A* 기반)

```python
class PathFinder:
    def find_path(
        self,
        start: tuple,
        end: tuple,
        grid: VoxelGrid,
        turn_count_limit: int,              # 진공:6, 배기:100 — 확정(◎)
        min_straight_distance: float,       # 진공:150mm, 배기:0 — 확정(◎)
        bending_optimization_weight: float, # 진공:5, 배기:0 — 확정(◎)
        accuracy: float = 100.0,            # 가설(△): 종료점 snap 허용 반경(mm)
        timeout_sec: float = 30.0,          # 확정(◎): 탐색 제한 시간, 초과 시 None 반환
    ) -> list[tuple] | None
```

> `turn_angles`(대각 이동)와 `is_vacuum_pipes`는 Phase 1 구현에 없다.
> 진공/배기 분기는 호출자가 파라미터 값을 달리 넘기는 방식으로 처리한다 (`bending_weight` 등).

**A* 탐색 상태 공간 (State Space):**

단순 (x,y,z)가 아닌 6-tuple로 상태를 정의해야 한다.
`min_straight_distance`와 `turn_count_limit` 제약은 방향/이력 의존적이기 때문.

```python
State = (
    vx,                   # int — voxel 좌표 X
    vy,                   # int — voxel 좌표 Y
    vz,                   # int — voxel 좌표 Z
    dir_idx,              # int — 방향 인덱스 0~5 (축방향 6방향)
    turn_count,           # int — 방향 전환 누적 횟수
    straight_count,       # int — 마지막 전환 이후 직선 이동 복셀 수 (min_straight_voxels 에서 캡)
)
# straight_ok = (straight_count >= min_straight_voxels)
# Phase 1: min_straight_voxels=1 → straight_count ∈ {0, 1} (사실상 이진값)
```

**메모리 추정 (vaccum_test_1 기준):**
- voxel 공간: 예) 100×100×30 = 300,000 복셀
- 방향 6개 × turn_count 7단계(0~6) × straight_count 2단계(0~1, Phase 1 기준) = 84 상태/복셀
- 최악 오픈리스트 크기: ~25M 노드 → 노드당 약 120 byte → **약 3GB (허용 불가)**
- **Phase 1 실측:** 160개 엔티티 전체 탐색 0.7s. 최악 추정과 괴리는 A* 가지치기 효과.

**메모리 임계치 초과 시 다운그레이드 전략 (우선순위 순):**

| 단계 | 전략 | 조건 |
| ---- | ---- | ---- |
| 1 | 대각 이동 비활성화 (Phase 1 미구현, 추가 시 우선 제거) | 상태 수 4배 감소 |
| 2 | `straight_count` 캡 유지, visited = dict 대신 numpy 3D array | 해시 오버헤드 제거 |
| 3 | search window 제한: 현재 위치에서 반경 N 복셀 밖 노드 pruning | N 값은 실측 후 결정 |
| 4 | 탐색 실패 처리: 30초 timeout 초과 시 entity 라우팅 실패로 처리 | result_report에 TIMEOUT 기록 |

> 단계 3 이상 적용 시 `bending_weight` 비용함수 의미가 달라질 수 있음. 1→2 순서로만 적용.

**핵심 제약:**
- `turn_count_limit`: 방향 전환 횟수 초과 시 탐색 중단
- `min_straight_distance`: `straight_count < min_straight_voxels` 상태에서 꺾임 시도 시 노드 제거. 복셀 누적 카운트로 정확히 구현.
- `bending_optimization_weight`: g-score에 방향 전환 비용 추가
- `greedy_turn`: routing_option.json 확정값이나 현재 구현 미적용 — Phase 2 검증 대상

## 3. 비용함수 (Hookup 관점)

A* g-score에 추가되는 비용:

```
g(n→n') = 이동거리 + bending_weight × 방향전환_여부
```

| 배관 종류 | bending_weight | 의미 |
| --------- | -------------- | ---- |
| 진공 | 5 | 꺾임 1번 = 직선 5 복셀 비용 → dead volume 최소화 |
| 배기 | 0 | 굴곡 패널티 없음 → 경사 연속성이 우선 |

**진공 배관에서 bending_weight=5의 실제 의미:**
- 방향 전환 1회 = 추가 비용 5 (복셀 5개 직선과 동등)
- turn_count_limit=6과 함께 → 꺾임이 적은 짧은 경로 우선 선택
- Hookup 관점: 꺾임 = dead volume 증가 = 고순도 진공 라인 오염 리스크

**배기 배관에서 bending_weight=0의 이유:**
- 배기 라인 주 제약 = 경사 연속성 (중력 배수)
- 굴곡 자체보다 경로 방향이 항상 하향 경사인지가 중요
- conductance 손실(D³/L 비례)은 postprocessing 검증에서 별도 처리

## 4. 순차 라우팅 처리

```
entities = sort_pocs(entities)  # 우선순위 정렬 (직경 큰 것 먼저)

for entity in entities:
    path = PathFinder.find_path(entity.start, entity.end, grid, ...)
    if path is not None:
        entity["path"] = path
        # 완료된 경로 복셀을 즉시 장애물로 등록
        for point in path:
            voxel = grid.world_to_voxel(point)
            grid.mark_obstacle(bounding_box_around(voxel, entity.diameter))
```

다중 배관 동시 최적화는 NP-hard → 순차 우선순위 방식으로 처리.
