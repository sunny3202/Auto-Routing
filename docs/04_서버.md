# arithmetic-server — FastAPI 서버

레거시 `arithmetic.exe` 대체. 신규 Python FastAPI 서버.

---

## 1. 역할

- REST API 제공 (Client ↔ Server 통신)
- RuleSet 파이프라인 오케스트레이션
- `routing_core` Python import로 경로 탐색 실행
- 비동기 BackgroundTask로 장시간 라우팅 처리

---

## 2. Canonical API 표 (변경 금지)

이 표만 참조. 다른 곳에 API 경로 적지 않는다.

| Method | Path | 역할 | 요청 | 성공 응답 |
| ------ | ---- | ---- | ---- | --------- |
| POST | `/api/v1/routing/upload/input/{historyId}` | 파일 업로드 + 파이프라인 시작 | FormData: `pocZip`, `obstacleFile`, `metadata` | `{ "requestId": str }` |
| GET | `/api/v1/routing/progress/{type}/{requestId}` | 진행률 폴링 | `type`: "validation" / "preprocessing" / "routing" | `{ "progress": float }` |
| POST | `/api/v1/routing/report/result/{requestId}` | 결과 리포트 조회 | — | result_report JSON array |
| POST | `/api/v1/routing/cancel/{requestId}` | 작업 취소 | — | `{ "ok": true }` |

> **업로드 계약 (확정):** 파일 업로드는 `pocZip` 단수 — ZIP 아카이브 1개. ZIP 안에 시나리오별 1~2개 파일 포함. 서버가 `workdir/{requestId}/poc/`에 압축 해제 후 폴더 경로를 `input_vaildation_fn`에 전달한다. multipart 다중 파일 또는 단수 txt 파일 업로드는 지원하지 않는다.

## 4. 업로드 페이로드 명세

### `metadata` 필드 (JSON string, FormData 내 전송)

```json
{
  "target": "Vacuum",
  "stage": "1단계",
  "historyId": "abc123"
}
```

| 필드 | 타입 | 값 | 비고 |
| ---- | ---- | -- | ---- |
| `target` | string | `"Vacuum"` / `"Exhaust"` | 시나리오 판별 1차 분기 |
| `stage` | string | `"1단계"` / `"2단계"` | 시나리오 판별 2차 분기 |
| `historyId` | string | Client 생성 UUID | requestId = historyId |

### `pocZip` — 시나리오별 ZIP 내부 파일 구성 (코드 기준 확정값)

모든 시나리오에서 `input_vaildation_fn(point_of_connectors_folder, ...)` 형태로 폴더를 받는다.
ZIP을 서버가 `workdir/{requestId}/poc/` 에 압축 해제하여 전달.

| 시나리오 | ZIP 내 필수 파일 | 코드 근거 |
| -------- | --------------- | --------- |
| 1단계 진공 | `Connectors.csv`, `poc_data.txt` | `create_entities(folder/Connectors.csv, folder/poc_data.txt)` 하드코딩 |
| 1단계 배기 | `equip_poc.txt` | `equip_poc_path = f"{folder}\\equip_poc.txt"` 하드코딩 |
| 2단계 진공 | `equip_poc.txt`, `pump_poc.txt` | equip 매핑 후 pump 파일 추가 로드 |
| 2단계 배기 | `equip_poc.txt` | 1단계 배기와 동일 함수 |

> **배기 1단계/2단계 파일 구조:** 둘 다 `equip_poc.txt` 단수. 파일 내용으로 구분 불가 → `metadata.stage` 필수.

### `obstacleFile` — BIM 장애물 정보 (JSON)

```json
{
  "bounding_box": [[xmin,ymin,zmin],[xmax,ymax,zmax]],
  "obstacles": [
    {"min": "X:-1000, Y:-1000, Z:-1000", "max": "X:1000, Y:1000, Z:1000"}
  ]
}
```

**진공→배기 의존성 — historyId 공유 방식 (◎ 확정):**

| 단계 | 실행자 | 동작 |
| ---- | ------ | ---- |
| 1. 진공 라우팅 | Client → Server | `historyId="abc"`로 진공 실행 → 완료 후 `workdir/abc/result.json` 생성 |
| 2. 배기 라우팅 | Client → Server | **동일 `historyId="abc"`**로 배기 실행 요청 |
| 3. 의존성 해소 | Server 자동 | 배기 preprocessing 시작 전 `workdir/abc/result.json`을 읽어 `bim_info["routing_result"]`에 삽입 |

→ **서버 구현:** 배기 파이프라인 시작 시 `workdir/{historyId}/result.json` 존재 확인. 없으면 `400 Bad Request` 반환.
→ **클라이언트 구현:** 진공 완료 폴링 확인 후 동일 `historyId`로 배기 실행. 순서를 보장해야 함.

## 5. 시나리오 판별 → RuleSet 경로 매핑

시나리오 판별은 `metadata.target + metadata.stage` 조합으로만 결정. `detect_scenario()`는 서버에서 1회만 정의.

```python
# 서버 내부 매핑 테이블 (변경 시 전체 RuleSet 재구성 필요)
RULESET_BASE = "AutoRouting_논리부 Rule-Set"
RULESET_MAP: dict[tuple[str, str], str] = {
    ("Vacuum",  "1단계"): f"{RULESET_BASE}/1단계 진공 배관 라우팅 input data",
    ("Exhaust", "1단계"): f"{RULESET_BASE}/1단계 배기 배관 라우팅 input data",
    ("Vacuum",  "2단계"): f"{RULESET_BASE}/2단계 진공 배관 라우팅 input data",
    ("Exhaust", "2단계"): f"{RULESET_BASE}/2단계 배기 배관 라우팅 input data",
}

def detect_scenario(metadata: dict) -> str:
    """metadata.target + metadata.stage → 시나리오 문자열"""
    target = metadata["target"]   # "Vacuum" | "Exhaust"
    stage  = metadata["stage"]    # "1단계" | "2단계"
    key = (target, stage)
    if key not in RULESET_MAP:
        raise ValueError(f"Unknown scenario: {key}")
    type_ = "진공" if target == "Vacuum" else "배기"
    return f"{stage}_{type_}"    # "1단계_진공" | "1단계_배기" | "2단계_진공" | "2단계_배기"

def get_ruleset_path(metadata: dict) -> str:
    """metadata → 해당 RuleSet 폴더 절대 경로"""
    return RULESET_MAP[(metadata["target"], metadata["stage"])]
```

## 6. 파이프라인 오케스트레이션

```python
async def run_routing_pipeline(request_id: str, poc_zip: bytes, bim_file: bytes, metadata: dict):
    scenario = detect_scenario(metadata)         # "1단계_진공" 등
    ruleset_path = get_ruleset_path(metadata)    # 서버 내부 매핑

    workdir = f"workdir/{request_id}"
    poc_folder = f"{workdir}/poc"
    extract_zip(poc_zip, poc_folder)             # ZIP 압축 해제

    bim_path = f"{workdir}/bim_info.json"
    save_file(bim_file, bim_path)

    ruleset = load_ruleset(ruleset_path)         # 해당 세트 Python 모듈 동적 import

    # 1. 입력 검증 — input_vaildation_fn은 폴더 경로를 받음
    entities, report = ruleset.input_vaildation_fn(poc_folder, bim_path, workdir)
    update_progress("validation", request_id, 100)
    save_json(f"{workdir}/input_validation_report.json", report)

    # 2. 전처리
    entities_path = f"{workdir}/point_of_connectors.json"
    entities, prep_report = ruleset.preprocessing_fn(entities_path, bim_path, workdir)
    update_progress("preprocessing", request_id, 100)
    save_json(f"{workdir}/preprocessing_report.json", prep_report)

    # 3. 라우팅 (routing_core) — SmartRoutingAI 대체
    from routing_core import RoutingPipeline
    pipeline = RoutingPipeline()
    entities = pipeline.run(entities, obstacles, routing_option)
    update_progress("routing", request_id, 100)

    # 4. 후처리
    # 현재 RuleSet: 진공=경로단순화, 배기=pass-through
    # Phase 4에서 validate_hookup_rules() 추가 예정 (신규 기능, 현재 미구현)
    entities = ruleset.postprocessing_fn(entities, obstacles, routing_option)

    # 5. 결과 저장
    save_json(f"{workdir}/result.json", entities)
    save_json(f"{workdir}/result_report.json", build_report(entities))
```

### 디스크 폴더 구조 (requestId별 격리)

```
arithmetic-server/
  workdir/
    {requestId}/
      poc/                         — ZIP 압축 해제 위치 (input_vaildation_fn에 전달)
        Connectors.csv             — 1단계 진공만
        poc_data.txt               — 1단계 진공만
        equip_poc.txt              — 배기/2단계 진공
        pump_poc.txt               — 2단계 진공만
      bim_info.json                — obstacleFile (배기 preprocessing이 routing_result 삽입)
      point_of_connectors.json    — input_vaildation 출력 (preprocessing 입력)
      input_validation_progress.json
      input_validation_report.json
      preprocessing_progress.json
      preprocessing_report.json   — 배기: 진공 POC 충돌 대상 목록 포함
      routing_progress.json
      result.json                  — 최종 경로 출력
      result_report.json           — 성공/실패 리포트
```

## 7. progress.json 포맷

```json
{ "progress": 75.3 }
```

- 범위: 0.0 ~ 100.0
- 각 단계 완료 시 100으로 갱신
- Client는 이 파일을 폴링 (`GET /progress/{type}/{requestId}`)
