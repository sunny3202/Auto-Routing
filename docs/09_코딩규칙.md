# 코딩 규칙 및 개발 계약

---

## 1. 절대 변경 금지 항목

| 항목 | 변경 금지 이유 |
| ---- | -------------- |
| `result.json` 스키마 | SARAI Addin VImportApp/RImportApp이 직접 참조. `path[-1]`, `path[-2]` 하드 접근 |
| Canonical API 4개 경로 | Client 코드가 경로 하드코딩 참조 |
| 알고리즘 행동 계약 표 값 | RuleSet 실측값. 변경 시 골든 샘플 실패 |
| Python 패키지명 `routing_core` | arithmetic-server `import routing_core` 참조 |
| 서버 포트 `localhost:8010` | Client next.config.js rewrites 하드코딩 |

## 3. 좌표 단위 규칙

| 위치 | 단위 | 변환 방식 |
| ---- | ---- | --------- |
| routing_core 내부 | mm | 기준 단위 |
| result.json `path` 배열 | mm | SARAI Addin이 `/ 304.8` 변환 |
| RuleSet POC 데이터 | mm | 입력 기준 |
| RevitDataExporter 출력 JSON | mm | `ToMm()` 적용 완료 |
| Revit API 내부 | feet | `UnitUtils.ConvertFromInternalUnits()` 필수 |

**주의:** RevitDataExporter의 `LocationPoint` 좌표는 `ToMm()` 적용 완료.
새 요소 추가 시 반드시 `ToMm()` 통과 확인.

## 4. 테스트 원칙

- 골든 샘플(vaccum_test_1, exhaust_test_1 등)을 mock으로 대체하지 않는다
- 각 Phase 종료 전 vaccum_test_1 성공률 수치 측정 필수
- result.json 스키마 변경 시 VImportApp 파싱 단위 테스트 동시 수정
- 배기 라우팅 테스트 전 진공 result.json 선행 생성 필수
- 경로 충돌 검사: `VoxelGrid.is_blocked()` 전수 확인 (샘플링 금지)

## 5. Hookup 검증 관련 규칙 — **[Phase 4 구현 예정]**

> **현재 미구현.** `validate_hookup_rules()` 함수는 Phase 4에서 신규 작성 예정.
> 현재 postprocessing은 경로 단순화(진공) 또는 pass-through(배기)만 수행한다.
> Phase 1~3에서는 아래 규칙이 적용되지 않으며, result_report에 `design_quality` 필드도 출력되지 않는다.

**아래는 Phase 4 구현 시 반드시 지킬 설계 규칙이다. 현재 코드에 적용 중이지 않다.**

- Phase 4 구현 시: `validate_hookup_rules()` 반환 `status`가 "ERROR"인 엔티티는 result.json에 포함하되 Client에서 빨간색 표시
- Phase 4 구현 시: "WARNING" 상태는 설계자가 현장 판단 가능 → 포함하고 주황색 표시
- Phase 4 구현 시: Hookup 검증 파라미터(경사 임계값, min_straight 등)는 routing_option.json에 추가하여 하드코딩 금지
